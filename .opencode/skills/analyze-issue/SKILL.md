# 分析 APISIX Issue

## 目标

为 Apache APISIX 提供一个一致、可追溯且可审查的 Issue 分析工作流。优先识别真实根本原因，并与 APISIX 的官方约定保持一致。


## 执行边界

默认模式为纯分析模式：

- 不要修改仓库文件或提交代码更改。
- 除非用户明确要求，否则不提供可直接应用的补丁实现。
- 如果用户要求提供实现，请先完成 Issue 分析，并明确声明执行模式已从分析模式切换到实现模式。
- 使用 GitHub API 获取所有 issue 相关信息，确保审查的全面性和准确性。
- 本地文件访问范围严格限制在 `~/apisix` 目录内；禁止访问用户目录下的其他文件或文件夹。

## 信息来源策略

仅使用以下来源：

- Apache APISIX 官方文档 (https://apisix.apache.org/docs/apisix/)
- Apache APISIX 官方仓库代码和测试 (https://github.com/apache/apisix)
- 目标 GitHub Issue 的内容（正文、评论以及同一仓库中的关联 PR）。

禁止使用博客、第三方教程或论坛帖子作为证据。

## 快速分类通道

首先通过以下三个问题进行初步分类，并记录一个临时类型：

1.  能否使用提供的 `apisix version`、`uname -a`、`openresty -V`、`etcd version`、复现步骤和日志来复现该行为？
2.  预期的行为是否在 APISIX 官方文档中有明确说明？
3.  仓库代码/测试是否证实了与文档预期不符？

分类决策：

- 大部分是问答 -> `question`
- 配置错误或不支持的用法 -> `question` / `invalid`
- 预期与实际行为之间存在可复现的差异 -> `bug`
- 旨在实现新功能或行为演进 -> `enhancement` / `feature-request`

## 受理工作流

1.  从用户输入中识别 Issue 编号。
2.  使用 GitHub API 获取 Issue 的所有相关信息。

## 最低证据标准

在进行根本原因分析之前，请核实 Issue 模板 (`bug_report.yml`) 中的信息：

- APISIX 版本 (`apisix version`)
- 操作系统 (`uname -a`)
- OpenResty / Nginx 版本 (`openresty -V` 或 `nginx -V`)
- etcd 版本 (`curl http://127.0.0.1:9090/v1/server_info`)
- 最小复现步骤
- 当前行为 vs. 预期行为
- 错误日志

如果缺少任何必要项，则分类为 `info-needed`，并且在给出明确的根本原因结论前停止。

## 拓扑检查

在进行根本原因分析之前，务必记录拓扑结构：

- APISIX 部署模式：传统模式、docker、kubernetes (apisix-helm)
- 配置中心：etcd 等

如果拓扑结构未知，应降低结论的可信度，并首先请求缺失的信息。

## 分析方法（先分类）

1.  根据 Issue 正文和评论确认报告的行为。
2.  根据官方文档确认预期行为。
3.  根据仓库代码和测试确认实际行为。
4.  首先根据标签列表对 Issue 类型进行分类：
    - `question`
    - `bug`
    - `enhancement`
    - `feature-request`
5.  如果需要更改行为，请解释其范围和兼容性影响。

务必在提出建议前完成根本原因分析。

## 证据方法

对于每个 Issue：

- 区分**观察**（直接观察到的事实）和**推断**（推理得出的结论）。
- 明确标记推断。
- 每个结论必须至少绑定一个可追溯的来源（参见信息来源策略）。
- 如果证据冲突，明确说明冲突，避免强行给出确定性结论。
- 对关键陈述使用稳定的证据 ID：
  - `OBS-<n>` 用于直接观察到的事实。
  - `INF-<n>` 用于推断。
- 每个 `INF` 必须引用一个或多个 `OBS`。
- **问题结论**中的每个结论必须至少引用一个证据 ID。
- 在每个 `OBS` 附近附上来源 URL/路径。
- 对每个关键结论，输出**可信度**：高 / 中 / 低。
- 如果可信度为低，不要给出硬性结论；切换到请求缺失信息的流程。

## 冲突解决规则

当证据发生冲突时，按以下顺序处理：

1.  官方文档定义了预期的行为边界。
2.  仓库代码/测试定义了当前的实际行为。
3.  Issue 陈述/评论描述了报告的症状。

如果文档和代码冲突：

- 当代码违反文档行为时，推断为**Bug**。
- 当代码是故意为之但文档过时/不清晰时，推断为**文档缺陷**。务必将此标记为推断，并引用两个来源。

## 类型和标签建议

在最终结论之前，提供 Issue 类型和标签的建议：

- **问题**：建议 `question`
- **误解 / 无效用法**：建议 `question`、`can't fix by us` 或 `can't reproduce`
- **Bug**：建议 `bug`，并可选地附带模块标签（例如 `plugin`、`ingress controller`、`performance`）。
- **增强 / 功能请求**：建议 `enhancement` 或 `feature-request`，并可选地附带 `help wanted` 或 `good first issue` 以邀请社区贡献。

当类型为 `bug` 或 `enhancement` 时，如果证据充分，添加模块/组件标签：

- 插件相关 -> `plugin`
- 核心路由/负载均衡 -> 核心模块（在结论中注明）
- Admin API -> admin 模块（在结论中注明）
- Ingress Controller -> `ingress controller`
- 性能问题 -> `performance`
- 文档问题 -> `doc`

如果模块归属不明确，首先只使用类型/状态标签。

## 强制输出结构

**四段式结构**（问题、误解 / 无效用法）：

1.  **问题理解**
2.  **根本原因**
3.  **问题分析**
4.  **问题结论**

**五段式结构**（Bug、增强、功能请求）：

1.  **问题理解**
2.  **根本原因**
3.  **问题分析**
4.  **代码级设计建议**（如适用）
5.  **问题结论**

在**问题结论**的末尾，附加：

- **证据可信度**：高/中/低
- **影响范围**：...
- **拓扑结构**：...
- **Issue 类型**：...
- **建议标签**：...
- **下一步行动**：...

## 缺失信息处理

如果证据不足，不要猜测。明确列出缺失的细节并请求提供，例如：

- APISIX 版本和部署模式
- 最小可复现配置和步骤
- 预期结果 vs. 实际结果
- 错误堆栈跟踪和完整的日志片段

当分类为 `info-needed` 时：

- 在一个合并的列表中请求最少的缺失证据。
- 设置一个 7-14 天的跟进窗口。
- 如果在窗口期后没有回应，建议使用 `no response` 标签关闭。

## 文档和代码引用规则

- 文档引用必须包含具体的 URL。
- 代码行为引用必须包含具体的仓库路径或函数名称。
- 所有引用必须遵守**信息来源策略**。
- 如果包含 Lua 示例，请使用 `lua` 围栏代码块。
